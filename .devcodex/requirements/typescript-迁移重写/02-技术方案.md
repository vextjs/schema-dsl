## 1. 方案概述

### 1.1 核心策略

本次迁移采用**分阶段全量重写**策略（而非渐进式改造），理由：

1. DslBuilder/DslAdapter 的解析逻辑深度耦合且存在语义不一致，渐进改造无法根除问题
2. TypeScript strict 模式要求所有模块同步迁移，否则类型边界无法收敛
3. 全量重写可在重写过程中自然消除 15 个高影响 Bug，无需额外打补丁

**开发分支**：`v2`（不影响 `main` 分支）

**输出方向**：

```text
schema-dsl/
├── src/          ← TypeScript 源码（新建）
├── dist/
│   ├── esm/      ← ESM 输出（tsc 直接输出）
│   └── cjs/      ← CJS 输出（构建脚本转换）
├── test/         ← vitest 测试（重写 v1 全部用例）
├── tsconfig.json ← 新建
└── package.json  ← 更新（版本/引擎/exports/scripts）
```

### 1.2 关键架构决策

| 决策 | 选择方案 | 备选方案 | 选择理由 |
|------|---------|---------|---------|
| 核心变革 | `DslBuilder` + `DslAdapter` 的解析逻辑合并为统一管线（TypeRegistry → DslParser → ConstraintParser → SchemaCompiler） | ① 渐进式改造（DslAdapter 逐步委托 DslBuilder） ② 仅修复 Bug 不重构管线 | 渐进式无法根除语义分歧（`string:N` 在两引擎中行为不同）；仅修复 Bug 不解决重复代码根因。统一管线是消除两套引擎语义分歧的唯一彻底方案 |
| 模块组织 | `src/` 取代 `lib/`，目录结构与 v1 基本对应但新增 `parser/` 子目录 | 完全重新设计目录结构（按功能域划分） | 与 v1 对应的目录降低迁移认知成本；仅新增 `parser/` 反映核心架构变化 |
| 类型定义 | `src/types/` 集中存放所有共享接口/类型，自动生成 d.ts | ① 各模块内联类型 ② 保留手写 d.ts | 集中类型利于跨模块复用和一致性维护；手写 d.ts 已证明会漂移（v1 教训） |
| 构建工具 | `tsc` + `tsup`（ESM 主力 + CJS 封装） | ① 纯 tsc（ESM+CJS 双配置） ② unbuild ③ rollup | tsup 底层 esbuild 构建速度快；原生支持 dual-mode 输出；纯 tsc 双配置维护成本高；unbuild 社区较小 |
| 测试框架 | `vitest` | ① 保留 mocha + chai ② jest | vitest 兼容 describe/it/expect 语法（迁移成本低）；原生 ESM 支持；jest ESM 支持不成熟 |
| 语言包 fallback | 缺失 key 自动降级到 `en-US` | ① 编译时强制所有语言包 key 完整（无 fallback） ② 降级到 key 名称本身 | 编译时强制虽安全但翻译工作量大且阻塞开发；降级到 key 名称用户体验差；en-US fallback 兼顾安全性和实用性 |
| 缓存基础设施 | `cache-hub`（工作区内部包，MemoryCache LRU+TTL 引擎 + schema-dsl 适配器委托包装） | ① 自行实现（v1 方案：LRUCache.ts + CacheManager.js） ② `lru-cache` 三方开源包 | 自行实现已存在 TTL 边界 Bug 且长期维护成本高；`lru-cache` 引入外部三方依赖、增加安全审计面且 API 需额外适配；cache-hub 为工作区内部包，API 与 schema-dsl 深度定制（构造参数直接对应），零额外外部依赖 |

## 3. 核心架构设计

### 3.1 统一 DSL 解析管线

这是本次重写最重要的架构变化，用于消除 v1 中 DslBuilder 与 DslAdapter 的双重解析引擎。

```text
输入
  │
  ├─ string: 'string:6:required'
  │         └─ DslParser.parseString()
  │               ↑ 预处理步骤（先于主解析）：
  │               ① /^array![\d-]/ → 替换为 'array:...' + required=true（v1 兼容语法）
  │               ② 末尾 '!' → required=true，剥离后再解析
  └─ object: { name: 'string:50!', age: 'number:0-120' }
            └─ DslParser.parseObject()
                   │ 遍历字段
                   └─ 字段值 → DslParser.parseString()
  │
  ▼
TypeRegistry.resolve(typeName)
  ├─ 内置类型（33 个）→ baseSchema
  ├─ 自定义类型（registerType）→ customBaseSchema
  └─ 未知类型 → warn + { type: 'string' }
  │
  ▼
ConstraintParser.parse(constraintStr, baseType)
  ├─ 修饰符：! = required, ? = optional
  ├─ 范围：0-100 = { minimum:0, maximum:100 }（支持负数）
  ├─ 单值：6 = exactLength:6（string）/ minimum:6（number）
  ├─ 枚举：foo|bar = { enum:['foo','bar'] }
  └─ 返回 Partial<JSONSchema>（绝不返回字符串）
  │
  ▼
SchemaCompiler.compile(baseSchema, constraints, meta)
  ├─ 合并 baseSchema + constraints
  ├─ 注入 _customMessages（特殊类型）
  ├─ 触发 PluginManager.hook('afterParse', schema)
  └─ 返回 JSONSchema
  │
  ▼
输出：标准 JSONSchema 对象
```

#### TypeRegistry（类型注册表）

解决 v1 三处类型列表不一致问题（`_internalKeys`、`_isKnownType`、`hasType`）：

```text
TypeRegistry
├── 内置类型注册表（Map<string, TypeDefinition>）
│   ├─ string, number, integer, boolean, array, object, ...（33 个）
│   └─ 每个类型携带：baseSchema, patterns?, customMessages?, isInternal?
├── 自定义类型注册表（Map<string, TypeDefinition>）
├── TypeRegistry.register(name, def)   ← DslBuilder.registerType() 委托到此
├── TypeRegistry.resolve(name)         ← 内置优先，自定义覆盖
└── TypeRegistry.getInternalKeys()     ← 替代 v1 的 _internalKeys 数组
```

**关键改进**：
- `multipleOf` 从 internal keys 中移除（修复 DB-01：toJsonSchema 丢失 multipleOf）
- `slug` 纳入标准类型列表（修复 DB-02）
- `patterns` 通过 TypeRegistry 统一注入（修复 DA-01：DslAdapter ReferenceError）

#### ConstraintParser

```text
ConstraintParser.parse(constraintStr, baseType): Partial<JSONSchema>

约束字符串格式：
  [modifier][typeConstraint]
  modifier = '!' (required) | '?' (optional)
  typeConstraint:
    'N'       → string: exactLength:N  /  number|array: maximum:N（保持 v1 语义：单值=上界）
    'N-M'     → minimum:N, maximum:M  （N/M 均支持负号和小数，正则 /^(-?\d*\.?\d*)-(-?\d*\.?\d*)$/）
    'x|y|z'   → enum:['x','y','z']
    '0-'      → minimum:0（只设下界，不设上界）
    '-100'    → maximum:100（只设上界；'-N' 中 N 取绝对值；⚠️ 与负数不同，负数下界用 '>=-N'）
    '>N' '>= N' '<N' '<=N' '=N' → 比较运算符（number/integer 专用，优先级最高）

  string 类型 '-N'：→ maxLength:N（与 number 对齐，取绝对值作为上界）

返回类型：Partial<JSONSchema>（从不返回 string）
错误处理：无法解析时返回 {}（不污染目标 schema）
```

#### SchemaCompiler

负责将各部分组装为最终 JSONSchema，并正确处理 `_internalKeys` 的清洗（`toJsonSchema()` 时）：

```typescript
interface SchemaCompiler {
  compile(type: TypeDefinition, constraints: Partial<JSONSchema>): JSONSchema
  toJsonSchema(schema: JSONSchema): JSONSchema  // 清除 internal keys，保留 standard keys
}
```

### 3.2 DslBuilder（精简后的链式 API 层）

v2 的 DslBuilder 不再包含解析逻辑，仅维护链式调用状态和委托到解析管线：

```text
DslBuilder（链式 API）
├── 构造：new DslBuilder(typeOrString)
│         └─ 调用 DslParser.parseString() 获得初始 schema
├── 每个链式方法（.min(), .max(), .email(), ...）
│   └─ 操作内部 _schema 对象（直接设置对应 JSONSchema 字段）
├── .toSchema()   → 返回含 _internal keys 的完整 schema
├── .toJsonSchema() → 返回清洗后的标准 JSONSchema（SchemaCompiler.toJsonSchema）
└── static DslBuilder.registerType() → 委托 TypeRegistry.register()
```

### 3.3 DslAdapter（精简薄适配层）

v2 的 DslAdapter 仅负责对象定义的字段分发，字符串解析统一委托 DslParser：

```text
DslAdapter（对象适配层）
├── parse(def: DslDefinition): JSONSchema
│   ├─ 遍历对象字段
│   ├─ 字符串字段 → DslParser.parseString()（消除 v1 重复实现）
│   ├─ 对象字段 → 递归 parse()
│   └─ DslBuilder 字段 → 直接调用 .toSchema()
├── static match(pattern, ...cases): JSONSchema  ← 保持 v1 行为
└── static if(condition): ConditionalBuilder | FieldCondition  ← 智能分发保持
```

### 3.4 Validator（增强类型安全）

```text
Validator
├── 构造：new Validator(options?: ValidateOptions)
│   ├─ 过滤非 AJV 选项（修复 V-Y01）
│   ├─ 创建 AJV 实例（仅一次）
│   └─ 注册 ajv-formats + CustomKeywords
├── validate(schema, data, opts?) → ValidationResult
│   └─ 内置 smartCoerceTypes（从 index.js 迁入）
├── validateAsync(schema, data, opts?) → Promise<ValidationResult>
│   └─ 内置 smartCoerceTypes（v1 的 validateAsync 缺此能力，v2 补齐）
├── static quickValidate(schema, data) → boolean
│   └─ 复用实例 AJV（修复 V-Y07）
└── _removeAdditional(schema, data)
    └─ 缓存内部 AJV 实例（修复 V-Y03）
```

> **入口 validate / validateAsync 函数规格**（src/index.ts 级别，区别于 Validator 类方法）：
>
> ```text
> validate(schema, data, opts):
>   ① _isDslObject(schema) → DslParser.parseObject(schema) 自动转换（保持 v1 行为）
>   ② opts.coerce !== false → smartCoerceTypes(data, schema)
>   ③ 复用单例 Validator 实例（getDefaultValidator()）
>   ④ 委托 Validator.validate(schema, data, opts)
>
> validateAsync(schema, data, opts):
>   ① _isDslObject(schema) → DslParser.parseObject(schema) 自动转换
>   ② opts.coerce !== false → smartCoerceTypes(data, schema)
>   ③ 复用单例 Validator 实例
>   ④ 委托 Validator.validateAsync(schema, data, opts)
> ```
>
> ⚠️ v1 中存在两套 validate 实现（index.js 的单例版 vs DslAdapter.js 的每次 new 版），v2 统一为 index.ts 的单例版，并补齐 validateAsync 的 smartCoerceTypes 能力。

**条件验证修复（V-02）**：

```text
_validateWithConditionals(schema, data):
  删除条件字段时：
    ✅ delete cleanSchema.properties[field]
    ✅ cleanSchema.required = cleanSchema.required?.filter(r => r !== field)  ← 新增
```

### 3.5 语言包 Fallback 机制

```typescript
// locales/types.ts — TS 强制 key 完整性
export type LocaleKey = keyof typeof zhCN  // zh-CN 为基准

// locales/index.ts — fallback 机制
function getMessage(key: LocaleKey, locale: string): string {
  const pack = getLocalePack(locale)
  const fallback = enUS  // en-US 作为 fallback
  const raw = pack[key] ?? fallback[key] ?? key  // 三级降级
  return typeof raw === 'object' ? raw.message : raw  // 统一返回 string
}
```

### 3.6 StringExtensions（opt-in）

数组驱动安装/卸载（修复 install/uninstall 不对称问题）：

```typescript
// 统一数组定义，确保对称
const EXTENSION_METHODS: Array<{
  name: keyof String
  fn: (...args: unknown[]) => unknown
}> = [
  { name: 'pattern',     fn: ... },
  { name: 'label',       fn: ... },
  { name: 'messages',    fn: ... },
  { name: 'format',      fn: ... },   // ← v1 安装但忘记卸载（S-01）
  { name: 'phone',       fn: ... },
  { name: 'phoneNumber', fn: ... },   // ← v1 安装但忘记卸载（S-02）
  // ... 其余方法
]

export function installStringExtensions(dsl: typeof dslFn): void {
  EXTENSION_METHODS.forEach(({ name, fn }) => {
    Object.defineProperty(String.prototype, name, { value: fn, configurable: true })
  })
}

export function uninstallStringExtensions(): void {
  EXTENSION_METHODS.forEach(({ name }) => {
    delete (String.prototype as Record<string, unknown>)[name]
  })
}
// 注意：v2 的 index.ts 不再自动调用 installStringExtensions()
```

> ⚠️ **BC-02 兼容性**：`StringExtensions.ts` 内部的 `installStringExtensions` 函数签名带 `dsl` 参数（用于在扩展方法中访问 dsl 实例）。但 v1 公开 API `installStringExtensions()` 无参数（`index.d.ts` 声明为 `(): void`）。
> **`index.ts` 必须以柯里化形式重新导出**：
> ```typescript
> // src/index.ts
> import { installStringExtensions as _install } from './core/StringExtensions.js'
> export const installStringExtensions = () => _install(dsl)
> ```
> 不得直接 re-export 内部函数，否则公开 API 签名将变为需要参数，违反 BC-02 承诺。

### 3.7 PluginManager（集成到核心管线）

v2 的 PluginManager 在 DslParser 和 Validator 中实际触发钩子：

```text
钩子触发点：
├─ DslParser.parseString()
│   ├─ beforeParse(input: string, ctx)
│   └─ afterParse(schema: JSONSchema, ctx)
├─ Validator.validate()
│   ├─ beforeValidate(schema, data, opts)
│   └─ afterValidate(result, schema, data, opts)
└─ SchemaCompiler.compile()
    ├─ beforeCompile(type, constraints)
    └─ afterCompile(schema)

异步钩子修复（CORE-08）：
  _runHook(name, ...args): Promise<void>
  await Promise.all(hooks.map(fn => fn(...args)))
```

**实例化策略（单例 + 依赖注入）**：

```text
// 模块级单例（默认实例）
export const pluginManager = new PluginManager()

// DslParser / Validator 构造函数接受可选注入

// 好处：
// - 生产环境：共享单例，全局插件注册一次生效
```

### 3.8 TemplateEngine（统一模板引擎）

合并 v1 中 `MessageTemplate.render()` 和 `ErrorFormatter._interpolate()` 两套不同逻辑：

```typescript
// src/core/TemplateEngine.ts
export function renderTemplate(template: string, params: Record<string, unknown>): string {
  // 单次替换（修复 CORE-03 模板注入漏洞）
  // 使用 Map 一次遍历，避免替换后的值被二次展开
  // 正则同时兼容两种格式：
  //   {{#key}}  ← v1 locale 文件（en-US/zh-CN 等）全部使用此格式
  //   {key}     ← v2 新增消息模板推荐格式
  let result = template
  result = result.replace(/\{\{#([^}]+)\}\}|\{([^}]+)\}/g, (match, k1, k2) => {
    const key = k1 ?? k2
    if (key in params) return String(params[key])
    return match  // 无对应 key 时原样保留，便于调试
  })
  return result
}
```

> ⚠️ **v1 locale 格式兼容**：v1 所有语言包（en-US、zh-CN 等）中的变量占位符均为 `{{#key}}` 格式。TemplateEngine **必须**同时支持此格式，否则所有错误消息中的变量（如 `{{#label}}`、`{{#min}}`）将静默保留原始占位符，导致用户看到未替换的 `{{#field}}` 字样。

|:-:|------|:----:|---------|
| P0-1 | `string:N` 语义统一为 `exactLength` | 🔴 已有使用 DslAdapter 对象语法 + `string:N` 的用户，行为由 maxLength 变为 exactLength | 在 CHANGELOG 中明确标注为行为修正（v1 两条路径本就不一致）；v1 单独分支保持不变；**紧急回退**：若发布后大量用户受影响，执行 `npm dist-tag add schema-dsl@1.x.x latest` 将 latest 指回 v1 |
| P0-2 | StringExtensions 默认不安装 | 🔴 使用 String.prototype 链式调用的用户（如 monSQLize examples）v2 运行时 TypeError | 迁移文档明确说明；examples/ 提供 opt-in 示例；**紧急回退**：`npm dist-tag add schema-dsl@1.x.x latest` 将 latest 指回 v1 |
| P0-3 | JSONSchemaCore 废弃 | 🟡 如果下游有第三方扩展 JSONSchemaCore 的用法 | 导出时加 `@deprecated` 注释，不直接删除导出；v1 分支保留 |
| P0-4 | Node >= 18 引擎要求 | 🟡 Node 16 用户无法使用 v2 | v1 分支维护（>=12），v2 明确标注 Node 18+ |
| P0-5 | DslBuilder.toJsonSchema() 输出格式 | 🔴 vext OpenAPI 管道依赖此方法，输出格式变化会破坏 OpenAPI 生成 | Phase 11 专项集成测试覆盖；对比 v1/v2 输出格式 diff。**回滚优先级**：① 发现不一致时优先修正 v2 实现以匹配 v1 输出 ② 若确认是 v1 Bug 导致的格式差异，则记录为行为修正并在 CHANGELOG 中声明 ③ 若修正成本过高，回退到 Phase 7 DslBuilder 重新设计 toJsonSchema() 输出逻辑 |

## §13 Breaking Changes 清单

> 本章节列出 v1 → v2 升级过程中对用户代码/配置/行为产生影响的所有变更。
> 按破坏等级分为三级：🔴 强制迁移 / 🟡 条件影响 / 🟢 特定场景。
> 对应 `templates/technical-design.md §6` 规范。

### 分级说明

| 级别 | 定义 |
|:----:|------|
| 🔴 强制迁移 | 必须修改代码/配置才能使用 v2，无论使用方式如何 |
| 🟡 条件影响 | 取决于使用方式，部分用户需修改 |
| 🟢 特定场景 | 仅影响 TypeScript 用户、废弃 API 用户等特定场景 |

---

### 🔴 强制迁移（必须修改才能使用 v2）

| # | ID | 变更内容 | 变更前（v1） | 变更后（v2） | 影响用户群 | 缓解措施 |
|:-:|:--:|---------|------------|------------|---------|---------|
| 1 | BC-01 | `string:N` DSL 语义统一为 `exactLength` | DslAdapter 路径：`dsl({ field: 'string:6' })` → `{ maxLength: 6 }` | 统一管线：`{ exactLength: 6 }` | 使用 `dsl()` 对象语法 + `string:N` 且依赖 maxLength 语义的用户 | 若需要 maxLength，改用 `string:0-N` 约束语法（如 `string:0-6`）；v1 分支保持 maxLength 语义不变 |
| 2 | BC-02 | StringExtensions 默认禁用，String.prototype 不自动扩展 | `require('schema-dsl')` 后 `.schema()` `.required()` 等 String.prototype 方法自动可用 | 默认不扩展，需显式调用 `installStringExtensions()` | 任何使用 `'type'.schema()` 等 String.prototype 链式调用的代码 | 在应用入口添加：`import { installStringExtensions } from 'schema-dsl'; installStringExtensions();` |
| 3 | BC-03 | Node.js 最低版本从 `>=12` 升至 `>=18` | `engines.node: ">=12.0.0"` | `engines.node: ">=18.0.0"` | 运行在 Node 12 / 14 / 16 的应用 | v1 分支继续维护（Node >=12）；建议升级到 Node 18 LTS |

---

### 🟡 条件影响（取决于使用方式，部分用户需修改）

| # | ID | 变更内容 | 变更前（v1） | 变更后（v2） | 影响条件 | 缓解措施 |
|:-:|:--:|---------|------------|------------|---------|---------|
| 4 | BC-04 | `lib/` 深路径导入不再有效 | `require('schema-dsl/lib/core/DslBuilder')` 可访问内部模块 | 源码移至 `src/`，构建输出到 `dist/`；无 `lib/` 路径 | 仅影响直接 import 内部模块路径的用户（使用了非公开 API） | 仅使用主入口 `require('schema-dsl')` / `import from 'schema-dsl'`；所有内部 API 已通过主入口正式导出 |
| 5 | BC-05 | 模块系统变为 ESM-first（`"type": "module"`） | CJS only（`module.exports`） | ESM-first + 同时提供 CJS 兼容层（`exports.require` → `.cjs`） | ① `require('schema-dsl')` **仍然兼容**（走 exports.require 条件导出）；② Node 旧版本 `--experimental-modules` 标志用法可能受影响 | 标准 `require()` 无需修改；如有 `--experimental-modules` 相关配置，升级到 Node 18 后移除 |
| 6 | BC-06 | `assert()` 错误类型变更（Bug 修复 C-03） | `I18nError.assert(cond, key)` 失败时抛出 `Error` 实例（v1 Bug） | 抛出 `ValidationError`（`ValidationError extends I18nError extends Error`） | 使用 `e.constructor.name === 'Error'` 或 `e.name === 'Error'` 精确匹配错误类型的 catch 块 | 改为 `e instanceof ValidationError` 或 `e instanceof I18nError`；`instanceof Error` 仍为 true，无影响 |

---

### 🟢 特定场景（仅影响特定用户群）

| # | ID | 变更内容 | 变更前（v1） | 变更后（v2） | 适用场景 | 缓解措施 |
|:-:|:--:|---------|------------|------------|---------|---------|
| 7 | BC-07 | TypeScript 类型声明完全重建（手写 → 自动生成） | 手写 `index.d.ts`，含错误声明（`ConditionalBuilder` 出现在 d.ts 但 `module.exports` 中未实际导出） | d.ts 由 tsc/tsup 自动生成，类型精确；`ConditionalBuilder` 正式导出 | TypeScript 用户若依赖了 v1 手写 d.ts 中的错误声明（如 `import { ConditionalBuilder } from 'schema-dsl'` 的类型注解） | v2 中 `ConditionalBuilder` 已正式导出，编译错误实为修复；更新导入路径即可 |
| 8 | BC-08 | `JSONSchemaCore` 标注 `@deprecated`（功能保留） | `JSONSchemaCore` 正常导出，无废弃标注 | 导出时标注 `@deprecated`，功能不变但不建议使用 | TypeScript 用户会看到 IDE deprecation 警告；直接扩展 JSONSchemaCore 的代码长期需迁移 | 短期内功能不受影响；建议迁移到统一解析管线（TypeRegistry + SchemaCompiler） |
| 9 | BC-09 | `dsl()` 返回类型签名改为函数重载（TypeScript Only） | v1 d.ts 中返回类型为联合类型 `DslBuilder \| JSONSchema` | v2 使用函数重载精确推断：传字符串返回 `DslBuilder`，传对象返回 `JSONSchema` | TypeScript 用户原本为处理联合类型而编写的 type narrowing 代码可能变为冗余（不报错，但多余） | 无需强制修改；冗余的类型收窄代码可逐步清理 |
| 10 | BC-10 | `cache-hub` 成为运行时生产依赖 | 无运行时外部依赖（LRUCache.ts 自实现，纯内部代码） | 新增 `cache-hub ^1.0.0` 运行时依赖 | 离线环境、bundle size 严格管控、安全审计要求零外部依赖的特定用户 | `cache-hub` 为工作区内部包，非 npm 公开包；离线用户需确认 cache-hub 可本地访问；bundle size 新增约 ~5KB（视构建工具 tree-shaking 效果而定） |

---

### 迁移指南摘要（v1 → v2）

1. **升级 Node.js**（BC-03）：将运行环境升级至 Node 18+。
2. **检查 `string:N` 语义**（BC-01）：全局搜索 `'string:\d+'` 模式，若使用 DslAdapter 对象语法且依赖 maxLength 行为，将其改为 `string:0-N`。
3. **添加 StringExtensions opt-in**（BC-02）：若项目中有 String.prototype 链式调用，在入口文件添加 `installStringExtensions()`。
4. **移除深路径导入**（BC-04）：将所有 `schema-dsl/lib/...` 路径改为从主入口导入。
5. **更新错误类型检测**（BC-06）：将 `e.constructor.name === 'Error'` 改为 `e instanceof ValidationError`。
6. **移除实验性 ESM 标志**（BC-05）：若项目使用了 `--experimental-modules` 或 `--es-module-specifier-resolution` 等旧 Node.js ESM 配置，升级到 Node 18 后将其移除（Node 18 原生支持 ESM，无需实验标志；标准 `require()` 调用无需任何修改）。

| 项 | 内容 |
|----|------|
| 维护分支 | `v1.x`（从 `main` 分支 v1.2.5 拉出） |
| 维护级别 | 仅安全修复（security-only）— 仅接受安全漏洞和导致数据损坏的关键 Bug 修复 |
| 维护期限 | v2.0.0 正式发布后 **6 个月**，届时 v1.x 进入 EOL（End of Life） |
| Node.js 支持 | 保持 `>=12.0.0`，不升级 |
| 功能冻结 | 不接受新功能、性能优化或非关键 Bug 修复 |
| 发布频率 | 按需发布（仅安全补丁时发版） |

## §14 原功能影响性评估

> 评估 v1 原有功能在 v2 完全重写后可能改变或静默失效的风险，补充 §4 Bug 修复设计和 §12 兼容清单之外的边缘行为视角，供实施者重点关注。

| 功能区域 | v1 行为 | v2 行为变化 | 风险等级 | 关联章节 |
|---------|---------|------------|:-------:|---------|
| `string:N` DSL 解析 | DslAdapter 路径→ `maxLength`；DslBuilder 路径→ `exactLength`（两条路径语义不一致） | 统一管线→ `exactLength`（有意行为修正） | 🔴 有意变更 | BC-01 / P0-1 |
| StringExtensions 自动安装 | `require('schema-dsl')` 后自动扩展 String.prototype | 默认不安装，需显式 `installStringExtensions()` opt-in | 🔴 有意变更 | BC-02 / P0-2 |
| exactLength/minLength/maxLength 计数 | 使用 `str.length`（UTF-16 字节单元，emoji 计数不正确） | 改用 `[...str].length`（Unicode 码点，emoji/surrogate pairs 正确计数） | 🟡 行为修正 | CK-Y04 / N10 |
| IPv4/IPv6 format 验证 | IPv4 正则过于宽松（接受非法地址）；IPv6 不支持压缩格式 | 替换为 RFC 合规标准正则（严格模式，旧宽松输入可能从通过变为拒绝） | 🟡 行为修正 | CF-01 / CF-02 |
| `CacheManager` 内部实现 | 自行实现（LRUCache.ts + CacheManager.js）；miss 返回 null | 委托 cache-hub（thin wrapper）；公开 API 不变，miss 语义须端对端验证 | 🟡 实现变更 | §8.4 / §12 F06 |
| `assert()` 抛出错误类型 | 抛出原生 `Error`（v1 Bug）| 抛出 `ValidationError`（正确行为，`instanceof Error` 仍为 true） | 🟡 行为修正 | BC-06 / C-03 |
| LRUCache 内部模块 | 可通过深路径 `lib/utils/LRUCache` 访问 | 完全移除，无任何替代深路径 | 🟡 条件影响 | BC-04 |
| 语言包缺失 key | 行为不定（可能返回 undefined 或 key 名） | 自动降级 en-US，行为可预期 | 🟢 改善 | §3.5 |
| PluginManager 独立实例化 | 可直接 `new PluginManager()` | 集成到核心管线，仍可通过统一接口访问；公开 API 兼容 | 🟢 内部重构 | §3.7 |

> ⚠️ **重点关注**：CacheManager miss 语义（cache-hub 委托后 null 返回行为）和 exactLength Unicode 计数行为是两个最需要端对端测试验证的边缘行为变化点（见 §12 兼容清单 + §10.3 CK-Y04 回归测试）。
