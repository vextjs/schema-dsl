## 1. 需求背景

### 问题描述

schema-dsl 当前版本（v1.2.5）为纯 JavaScript 实现，存在以下核心问题：

1. **类型安全缺失**：手写的 `index.d.ts`（~3650 行）与运行时实现长期脱节，下游 TypeScript 项目（vext、monSQLize）频繁遭遇类型不匹配问题
2. **严重的代码重复与语义分歧**：`DslBuilder.js`（~1587 行）与 `DslAdapter.js`（~988 行）之间存在大量重复的类型映射和 DSL 解析逻辑，且已确认存在关键语义不一致（`string:N` → DslBuilder 生成 `exactLength`，DslAdapter 生成 `maxLength`）
3. **全局副作用**：`StringExtensions.js` 在 `require()` 时自动修改 `String.prototype`，属于全局原型污染，对库的使用者构成隐患
4. **入口文件职责混杂**：`index.js` 承载了 i18n 加载、单例管理、智能类型转换、配置处理等多种职责，难以维护和测试
5. **模块系统落后**：虽然提供了 `index.mjs` 作为 ESM 入口，但核心代码仍为 CommonJS，无法利用 tree-shaking 等现代构建优化
6. **Node.js 最低版本过旧**：当前要求 `>=12.0.0`，无法使用现代 Node.js API
7. **大量已知缺陷**：三轮深度检查发现 41 个 🔴 Bug、56 个 🟡 待改进、39 个 💡 建议（合计 136 项），其中 15 个为高影响 Bug（会导致运行时错误或数据不一致）
8. **617 行死/空置代码**：`JSONSchemaCore`（282 行，无任何模块引用）+ `PluginManager`（335 行，9 个钩子但无模块触发）

### 期望目标

将 schema-dsl 完整重写为 TypeScript，产出 v2.0.0 版本，实现：

- 原生类型安全，自动生成 `.d.ts` 声明文件（消除手写 d.ts 漂移问题）
- 消除 DslBuilder/DslAdapter 之间的逻辑重复，建立统一的解析→编译管线
- **修复全部 15 个 P0/P1 高影响 Bug**（在重写过程中自然消除）
- String.prototype 扩展改为 opt-in（默认不安装）
- 模块化入口，职责分离清晰
- ESM 优先 + CJS 兼容双模式输出
- 100% 向后兼容下游项目（vext、monSQLize）的公开 API 契约

### 触发原因

| 项 | 内容 |
|----|------|
| 来源 | 技术债务 + 下游项目（vext）TypeScript 生态需求 + 三轮深度检查暴露的严重缺陷 |
| 紧迫度 | 🟡 正常 |
| 影响范围 | schema-dsl 全模块 + 下游 vext（schema-adapter）+ monSQLize（model 层） |

## 3. 关键决策记录

| D1 | `'string:N'` 单值约束统一语义 | 采用 `exactLength`（精确长度） | DslBuilder 的行为是更精确的语义；`maxLength` 可用范围语法 `0-N` 表达；迁移前两条路径语义不一致（DA-03），必须统一 | 2026-03-18 会话01 |

| D3 | Node.js 最低版本 | >= 18 | 使用现代 API（structuredClone、Intl.Segmenter 等）；Node 16 已 EOL | 2026-03-18 会话01 |
| D4 | 验证引擎 | 保持 AJV 8.x | 生态成熟，已有大量自定义 keyword 依赖 | 2026-03-18 会话01 |

| D6 | 死代码处理 | JSONSchemaCore 废弃，PluginManager 集成 | JSONSchemaCore 282 行无引用直接废弃；PluginManager 335 行有设计价值，重写时集成到核心管线 | 2026-03-18 会话02 |
| D7 | 模板引擎统一 | 合并 MessageTemplate 和 ErrorFormatter._interpolate | 两套模板替换逻辑使用不同正则和行为，TS 重写时统一为单一模板引擎 | 2026-03-18 会话02 |
| D8 | 导出清单一致性 | ConditionalBuilder 纳入导出 | v1 的 index.d.ts 声明了但 module.exports 未导出，v2 统一导出 | 2026-03-18 会话02 |
